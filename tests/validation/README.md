# Validation suite

Externalized validation for the slide-publisher pipeline. The harness here runs each test case through the relevant pipeline stages and lands artifacts in per-case folders. **Scoring is the user's job, not the harness's.** The harness is built to avoid self-affirming validation (passing tests that only confirm my code does what I designed it to do); scoring per the `EXPECTED.md` rubric is what actually validates the system.

## Layout

```
tests/validation/
├── README.md               ← this file
├── run_validation.py       ← harness
├── SCORESHEET.md           ← empty per-case scoring template
└── cases/
    ├── case-01-well-formed-template/
    │   ├── INPUT-spec.md
    │   ├── input.pptx
    │   ├── EXPECTED.md
    │   ├── 01-classifier.json     ← generated by harness
    │   ├── 02-extractor.json
    │   ├── 03-validator.json
    │   ├── 04-render.pptx
    │   ├── 04-render.pptx.loss.md
    │   └── 04-render.pptx.loss.json
    ├── case-02-weak-template/
    │   ├── INPUT-spec.md
    │   ├── input.pptx
    │   ├── EXPECTED.md
    │   ├── (per-stage outputs)
    │   ├── R-remediated.pptx       ← from remediation-apply
    │   ├── R-remediated.pptx.audit.md
    │   ├── R-remediated.pptx.audit.json
    │   └── R-revalidator.json
    ├── case-03-canonical-deck/     ← deck-with-implicit-pattern
    ├── case-05-mixed/
    ├── case-06-empty-template/
    ├── case-07-minimal-ir/         ← IR edge case
    └── case-08-invalid-ir/         ← IR edge case (validator should reject)
```

Three additional cases — **case-04 (real-world brand-heavy deck), case-09 (real Figma template), case-10 (real Figma deck-with-implicit-pattern)** — are intentionally not shipped in this public repo. They require your own real-world inputs; document them locally for your validation pass.

## Running

```bash
cd "<plugin-root>"
python tests/validation/run_validation.py
```

The harness processes each case directory and writes stage outputs to that directory. **It deliberately does NOT score outputs** — that's `SCORESHEET.md`'s purpose.

## Scoring

After the harness runs, open each case's `EXPECTED.md`. It walks you stage-by-stage through what to look for, what counts as pass, what counts as fail, and what the failure modes indicate diagnostically. Mark Y/N/Partial per check. Roll up into `SCORESHEET.md`.

## What the harness is NOT

- Not a CI gate. Use the integration harness at `tests/integration/test_pipeline.py` for that — it has self-checking assertions.
- Not a substitute for human inspection. The whole point of this suite is that the system is validated by external judgment, not by my assertions.
- Not exhaustive. Seven cases cover the canonical failure-mode matrix; real-world edge cases will surface ones we haven't anticipated. Score the failures honestly.
